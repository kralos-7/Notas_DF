WAI-ARIA · Comparativa de Accesibilidad

10 ejemplos lado a lado — sin vs con atributos de accesibilidad

Sin accesibilidad Con WAI-ARIA

Ejemplo 01 / 10

Asociación de etiquetas con campos

Un lector de pantalla necesita saber qué etiqueta pertenece a qué campo. Sin la asociación correcta, el usuario no sabe qué se le está pidiendo.

❌ Sin Label visualmente cerca, pero no asociado

Nombre

Correo

<!-- ❌ Usa <p> como etiqueta visual -->
<p>Nombre</p>
<input type="text" />

<p>Correo</p>
<input type="text" />

<!-- El lector de pantalla no sabe
     qué campo es qué -->
✅ Con Label asociado con for + id 
<!-- ✅ for="id" crea asociación semántica -->
<label for="nombre">Nombre</label>
<input type="text" id="nombre" />

<label for="correo">Correo</label>
<input type="email" id="correo" />

<!-- Click en el label enfoca el campo -->
🔍 ¿Por qué importa? Con <label for="id"> el lector de pantalla anuncia "Nombre, campo de texto" al enfocar el input. Sin esto, solo dice "campo de texto en blanco" y el usuario no sabe qué escribir. Además, el for hace que hacer clic en el label enfoque el campo, mejorando la usabilidad para todos.

Ejemplo 02 / 10

Mensajes de error accesibles

Un error visible visualmente puede ser completamente invisible para un lector de pantalla si no está vinculado al campo que lo provocó.

❌ Sin Error solo visual, no vinculado

⚠ Ingresa un correo válido

<label for="email">Correo</label>
<input type="text" id="email" />

<!-- ❌ El error existe en el DOM
     pero el input no sabe que existe -->
<p style="color:red">
  Ingresa un correo válido
</p>
✅ Con aria-describedby + aria-invalid

⚠ Ingresa un correo válido

<label for="email">Correo</label>
<input type="email" id="email"
       aria-invalid="true"
       aria-describedby="error-msg" />

<!-- ✅ role=alert anuncia el error
     automáticamente al lector -->
<p id="error-msg" role="alert">
  Ingresa un correo válido
</p>
🔍 ¿Por qué importa? aria-invalid="true" le dice al lector de pantalla que el campo tiene un error. aria-describedby vincula el campo con el mensaje de error por id, así el lector anuncia ambos juntos. role="alert" hace que el mensaje se anuncie automáticamente en cuanto aparece en el DOM.

Ejemplo 03 / 10

Campos requeridos

El asterisco (*) es una convención visual. Un lector de pantalla no sabe que significa "obligatorio" a menos que se lo indiquemos explícitamente.

❌ Sin Solo el asterisco visual 
<!-- ❌ El * es visual.
     El lector lo lee literalmente
     como "asterisco" o lo ignora -->
<label for="nombre">
  Nombre <span>*</span>
</label>
<input type="text" id="nombre" />
✅ Con required + aria-required + aria-hidden 
<label for="nombre">
  Nombre
  <!-- aria-hidden oculta el * al lector -->
  <span aria-hidden="true"> *</span>
</label>
<input type="text" id="nombre"
       required
       aria-required="true" />
🔍 ¿Por qué importa? required activa la validación nativa del navegador. aria-required="true" es la señal para lectores de pantalla — algunos no interpretan required solo. El aria-hidden="true" en el asterisco evita que el lector diga "asterisco" — el campo ya indica "requerido" semánticamente.

Ejemplo 04 / 10

Regiones en vivo (Live Regions)

Cuando el contenido cambia dinámicamente (contadores, notificaciones), un lector de pantalla no lo detecta a menos que esté marcado como región en vivo.

❌ Sin Contador visual, silencioso para lectores

0/100

<textarea id="msg"></textarea>

<!-- ❌ El contador cambia visualmente
     pero el lector de pantalla
     no sabe que el contenido cambió -->
<p id="contador">0/100</p>
✅ Con aria-live="polite"

0 de 100 caracteres usados

<textarea aria-describedby="contador">
</textarea>

<!-- ✅ aria-live anuncia el cambio
     aria-atomic lee el texto completo -->
<p id="contador"
   aria-live="polite"
   aria-atomic="true">
  0 de 100 caracteres usados
</p>
🔍 ¿Por qué importa? aria-live="polite" indica que la región se actualizará y el lector debe anunciarla cuando el usuario termine de leer lo que está leyendo. aria-atomic="true" hace que se lea el texto completo del elemento cada vez que cambia, no solo la parte que cambió.

Ejemplo 05 / 10

Barra de progreso

Una barra visual de progreso no comunica nada a usuarios de lectores de pantalla sin los atributos ARIA correctos.

❌ Sin Solo visual, sin semántica

Cargando archivo...

65% 
<!-- ❌ div decorativo sin info
     para lectores de pantalla -->
<div class="barra-wrap">
  <div class="barra-fill"
       style="width: 65%">
  </div>
</div>
✅ Con role="progressbar" + aria-valuenow

Cargando archivo...

65% 
<!-- ✅ Semántica completa para
     lectores de pantalla -->
<div role="progressbar"
     aria-valuenow="65"
     aria-valuemin="0"
     aria-valuemax="100"
     aria-labelledby="etiqueta"
     aria-valuetext="65% completado">
  <div style="width:65%"></div>
</div>
🔍 ¿Por qué importa? role="progressbar" define semánticamente qué es el elemento. aria-valuenow, aria-valuemin y aria-valuemax dan el valor actual y el rango. aria-valuetext permite dar una descripción más legible ("65% completado") en lugar del número solo.

Ejemplo 06 / 10

Botones de selección (Toggle)

Botones que tienen estado activo/inactivo deben comunicar ese estado más allá del color o estilo visual.

❌ Sin Estado solo con CSS, invisible semánticamente 
<!-- ❌ El estado .pressed es visual
     El lector no sabe qué está
     seleccionado -->
<button class="toggle-btn pressed">
  JavaScript
</button>
<button class="toggle-btn">
  CSS
</button>
✅ Con aria-pressed comunica el estado 
<!-- ✅ aria-pressed="true/false"
     comunica el estado al lector -->
<div role="group" aria-labelledby="lbl">
  <button aria-pressed="true">
    JavaScript
  </button>
  <button aria-pressed="false">
    CSS
  </button>
</div>
🔍 ¿Por qué importa? Sin aria-pressed, el lector anuncia "JavaScript, botón" sin indicar si está activo. Con aria-pressed="true" anuncia "JavaScript, botón activado". El role="group" con aria-labelledby agrupa los botones bajo la etiqueta "Categorías" para contexto adicional.

Ejemplo 07 / 10

Pestañas (Tabs)

Un patrón de tabs implementado con botones genéricos es visualmente funcional pero semánticamente vacío para tecnologías asistivas.

❌ Sin Tabs sin roles ni estado
Información de tu perfil público.
Cambia tu contraseña aquí.
Configura tus alertas.
<!-- ❌ Botones normales, sin rol
     El lector no sabe que son tabs -->
<div>
  <button class="activo">Perfil</button>
  <button>Seguridad</button>
</div>
<div>Info del perfil...</div>
<div class="oculto">Seguridad...</div>
✅ Con role="tablist/tab/tabpanel" + aria-selected
Información de tu perfil público.
Cambia tu contraseña aquí.
Configura tus alertas.
<div role="tablist" aria-label="Config">
  <button role="tab" aria-selected="true"
          aria-controls="panel-1"
          id="tab-1">Perfil</button>
</div>
<div role="tabpanel"
     id="panel-1"
     aria-labelledby="tab-1">
  Info del perfil...
</div>
🔍 ¿Por qué importa? El patrón completo role="tablist" → role="tab" → role="tabpanel" es reconocido por todos los lectores de pantalla. aria-selected="true/false" indica qué pestaña está activa. aria-controls vincula cada tab con su panel, y aria-labelledby hace la conexión inversa.

Ejemplo 08 / 10

Ventana modal / Diálogo

Un modal sin ARIA puede atrapar al usuario de teclado o lector sin posibilidad de saber que hay un diálogo abierto o cómo cerrarlo.

❌ Sin Modal sin rol ni gestión de foco

¿Eliminar cuenta?

Esta acción no se puede deshacer.

<!-- ❌ div sin semántica de diálogo.
     El foco no se mueve al modal.
     Esc no lo cierra. Sin etiqueta. -->
<div id="modal">
  <h3>¿Eliminar cuenta?</h3>
  <button>Eliminar</button>
  <button>Cancelar</button>
</div>
✅ Con role="dialog" + aria-modal + aria-labelledby

¿Eliminar cuenta?

Esta acción no se puede deshacer.

<div role="dialog"
     aria-modal="true"
     aria-labelledby="titulo"
     aria-describedby="desc">
  <h3 id="titulo">¿Eliminar?</h3>
  <p  id="desc">Sin deshacer.</p>
  <button>Eliminar</button>
  <button>Cancelar</button>
</div>
🔍 ¿Por qué importa? role="dialog" avisa al lector de pantalla que hay un diálogo activo. aria-modal="true" indica que el contenido detrás está bloqueado. aria-labelledby hace que el lector anuncie el título al abrir. En producción también se debe gestionar el foco (focus()) y capturar la tecla Escape para cerrar.

Ejemplo 09 / 10

Botones con solo icono

Un botón que solo muestra un ícono o emoji es completamente opaco para usuarios de lectores de pantalla sin una etiqueta accesible.

❌ Sin Icono solo, sin etiqueta

Acciones del archivo

El lector solo dirá "botón" sin contexto

<!-- ❌ El emoji no es texto legible
     de forma confiable por los
     lectores de pantalla -->
<button>✏️</button>
<button>📤</button>
<button>🗑️</button>
✅ Con aria-label + aria-hidden en el icono

Acciones del archivo

El lector anunciará "Editar archivo, botón"

<!-- ✅ aria-label describe la acción.
     aria-hidden oculta el icono
     al lector para evitar duplicados -->
<button aria-label="Editar archivo">
  <span aria-hidden="true">✏️</span>
</button>
<button aria-label="Eliminar archivo">
  <span aria-hidden="true">🗑️</span>
</button>
🔍 ¿Por qué importa? aria-label reemplaza completamente el contenido del botón para lectores de pantalla, dando un nombre descriptivo de la acción. El aria-hidden="true" en el icono evita que el lector lo duplique (sin él podría decir "lápiz, Editar archivo, botón"). Para íconos SVG es la misma técnica: aria-hidden="true" en el SVG y aria-label en el botón.

Ejemplo 10 / 10

Indicador de carga (Spinner)

Un spinner animado no comunica nada al usuario de lector de pantalla: no sabe si la página está cargando, si algo falló, o si ya terminó.

❌ Sin Animación pura, sin estado comunicado
Guardando cambios...

El lector de pantalla ignora este div completamente

<!-- ❌ div animado, invisible
     para lectores de pantalla -->
<div class="spinner"></div>
<span>Guardando cambios...</span>

<!-- No hay anuncio, el usuario
     no sabe qué está pasando -->
✅ Con role="status" + aria-live + texto sr-only
Guardando cambios... 

<!-- ✅ role="status" es un live region
     implícito (aria-live="polite") -->
<div role="status"
     aria-live="polite"
     aria-label="Guardando...">
  <div class="spinner"
       aria-hidden="true"></div>
  <span>Guardando cambios...</span>
</div>
<!-- aria-live="assertive" para errores -->
<p aria-live="assertive"></p>
🔍 ¿Por qué importa? role="status" es equivalente a aria-live="polite" y avisa al usuario que hay información de estado. El spinner recibe aria-hidden="true" porque es puramente decorativo. Para el resultado final (éxito/error), se usa aria-live="assertive" porque es información urgente que debe interrumpir la lectura actual del usuario.
